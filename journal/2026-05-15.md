# 2026-05-15 — Persistent Memory + Embeddings (M4)

## What was built
- `agent/memory.py` — `remember()` and `recall()` using `nomic-embed-text` via Ollama
- `memory` table in DuckDB with `FLOAT[]` embedding column
- Two new agent tools: `remember` and `recall`
- Similarity search using DuckDB's built-in `list_cosine_similarity()`

## Key decisions
- **DuckDB for vector storage** — avoids adding FAISS as a dependency. For personal memory (dozens to hundreds of facts) DuckDB's cosine similarity in pure SQL is fast enough.
- **nomic-embed-text** — local embedding model via Ollama, no external API calls
- **Hash-based memory ID** — `SHA256(fact)[:16]` deduplicates identical facts silently

## Problems faced
- **Agent calling too many tools after remember()** — after saving a fact the model kept calling `recall`, `get_latest_price`, `get_recent_activities`, and `get_health_summary` before hitting the 5-round limit without giving a final answer. Fixed by tightening the system prompt ("call only the tools needed, stop once you have enough information") and raising `MAX_TOOL_ROUNDS` to 8.

## Concepts touched
- Embeddings and vector similarity search
- RAG foundations — storing and retrieving unstructured facts alongside structured SQL data
- DuckDB array/list types and built-in vector functions
- Prompt engineering to constrain agentic behavior
